A firewall for
MCP servers.

mcpfw sits between your AI client and MCP servers, enforcing YAML-based policies that block credential theft, prompt injection, and billing amplification — at the wire level, before the model sees the response.

How it works

Four enforcement layers.

Each layer targets a different threat vector. No single layer covers all five attacks — the stack is necessity, not redundancy.

1
Tool name
allowed_tools whitelist · blocked_tools denylist
↓
2
Path check
allow_paths — canonical resolution, traversal-safe
↓
3
Argument content
block_patterns — regex on all flattened arg values
↓
4
Response DLP
detect_patterns — scans server responses before AI sees them

Quickstart

Wrap any MCP server in two steps.

Replace the server command in your MCP client config with mcpfw. No changes to the server, no changes to the client.

~/.mcpfw/policy.yaml 
# Generate a starter policy
$ mcpfw init

# ~/.mcpfw/policy.yaml (edit to taste)
version: 1
default_action: allow

response_rules:
  - name: detect-private-key
    detect_patterns:
      - '-----BEGIN (RSA |EC |OPENSSH )?PRIVATE KEY-----'
    action: block
    severity: critical

servers:
  - server: "filesystem-helper"
    allowed_tools: [read_file, list_directory]
    tool_rules:
      - name: restrict-paths
        tools: [read_file]
        allow_paths: [~/projects/myapp]
        severity: critical
~/Library/Application Support/Claude/claude_desktop_config.json 
// Before: direct server
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/alice"]
    }
  }
}

// After: wrapped with mcpfw
{
  "mcpServers": {
    "filesystem": {
      "command": "mcpfw",
      "args": ["wrap", "--policy", "~/.mcpfw/policy.yaml", "--",
               "npx", "-y", "@modelcontextprotocol/server-filesystem", "/Users/alice"]
    }
  }
}
